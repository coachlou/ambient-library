# Architecture

How Ambient Library works under the hood. For users who want to understand the system, debug it, or extend it.

## Overview

The system has three layers:

```
┌─────────────────────────────────────┐
│           Claude Session            │
│  Skills loaded transparently from   │
│  .agents/skills/ at session start   │
└────────────────┬────────────────────┘
                 │
┌────────────────▼────────────────────┐
│          .agents/skills/            │
│  Lightweight SKILL.md pointers,     │
│  generated by setup-skills.sh       │
│  (gitignored — never committed)     │
└────────────────┬────────────────────┘
                 │
┌────────────────▼────────────────────┐
│       skills/ (git submodule)       │
│  ambient-library — the source of    │
│  truth for all skill definitions    │
└─────────────────────────────────────┘
```

## Components

### ambient-library (this repo)

The central skill library. Contains:
- **Skill folders** (`skill-loader/`, `code-review/`, etc.) — each with a `SKILL.md` and `instructions.md`
- **`templates/`** — starter files for new projects
- **`bootstrap.sh`** — one-command project setup
- **`docs/`** — user documentation

Lives on GitHub. Projects reference it as a git submodule.

### skills/ (project submodule)

Each project contains `skills/` as a git submodule pinned to a specific commit of ambient-library. This means:
- Projects get a stable, reproducible snapshot of the skill library
- Updates are opt-in (`git submodule update --remote`)
- Multiple projects can run different versions simultaneously

### skills-manifest.yaml (per project)

Controls which skills are active for a given project:

```yaml
skills:
  - skill-loader        # required
  - skill-system-manager
  - code-review
```

Only skills listed here get installed into `.agents/skills/`.

### setup-skills.sh (per project)

Reads `skills-manifest.yaml` and copies `SKILL.md` from each skill in `skills/` into `.agents/skills/`. This generates lightweight pointers — not full copies — so the working directory stays lean.

Run manually or automatically via the session hook.

### .agents/skills/ (generated, gitignored)

The live skill pointers Claude reads during a session. Generated by `setup-skills.sh`. Never committed to git — always regenerated on demand.

### .claude/hooks/session-start.sh (per project)

Optional. Runs `setup-skills.sh` automatically at the start of every Claude session, keeping skill pointers fresh without any manual step.

### Skill Loader

The always-active orchestration skill. At session start it:
1. Confirms the skill system is ready
2. Resolves skill paths from `.agents/skills/`
3. Merges project rules from `CLAUDE.md` / `AGENTS.md` into each skill's context
4. Caches loaded skills for the session

This is why skills feel like built-in capabilities — the loader handles everything invisibly.

## Data Flow

**First-time setup:**
```
bootstrap.sh
  → git submodule add ambient-library → skills/
  → cp templates/* → project root
  → setup-skills.sh
      → reads skills-manifest.yaml
      → copies SKILL.md files → .agents/skills/
```

**Every session (with hook):**
```
Claude starts
  → .claude/hooks/session-start.sh
      → setup-skills.sh
          → refreshes .agents/skills/
  → Skill Loader activates
      → reads .agents/skills/
      → merges CLAUDE.md rules
      → skills are ready
```

**Updating skills:**
```
git submodule update --remote   # pull latest from ambient-library
./setup-skills.sh               # refresh pointers
restart Claude session          # pick up changes
```

## Design Decisions

**Why git submodules?**
Submodules give each project a pinned, reproducible snapshot of the skill library. Updates are explicit and intentional — no surprise breakage from upstream changes.

**Why SKILL.md pointers instead of full copies?**
Copying only `SKILL.md` keeps the project directory lean and makes it clear that the authoritative source is ambient-library, not the project. Full skill logic lives in `instructions.md`, which is loaded on demand.

**Why gitignore `.agents/skills/`?**
Generated artifacts don't belong in version control. The manifest + submodule together define what skills a project uses — that's enough to reproduce the state on any machine.

**Why a manifest instead of auto-discovery?**
Explicit is better than implicit. Projects should consciously choose which skills are active, not inherit everything in ambient-library automatically.

## Extending the System

- **New skill** → add a folder to ambient-library, commit, push. Projects opt in via manifest.
- **Project-specific rules** → edit `CLAUDE.md` in the project. Skill Loader merges them automatically.
- **Private skills** → create skill folders directly in the project (outside `skills/`), copy their `SKILL.md` into `.agents/skills/` manually, and reference them in `CLAUDE.md`.
