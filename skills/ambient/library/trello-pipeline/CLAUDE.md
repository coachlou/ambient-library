# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies
pip install anthropic python-dotenv

# Create the Trello board (run once, writes board_config.json)
python setup_board.py
python setup_board.py --name "My Pipeline"

# Trello board operations (used by Claude Code orchestrator)
python trello_ops.py status                          # Show board status (card counts per stage)
python trello_ops.py cards "Spec Review"             # List cards in a stage
python trello_ops.py card_detail <card_id>           # Full card with comments
python trello_ops.py comments <card_id>              # Card comments only
python trello_ops.py move <card_id> "Spec Review"    # Move card to stage
python trello_ops.py comment <card_id> "text"        # Add comment to card
python trello_ops.py create "Spec Write" "name" "desc"  # Create new card
python trello_ops.py add_label <card_id> "Approved"  # Add label
python trello_ops.py remove_label <card_id> "In Progress"  # Remove label
python trello_ops.py attach <card_id> <file_path>     # Attach file to card
python trello_ops.py submit "Your product idea"      # Submit idea to Intake
python trello_ops.py find_idea_brief <idea_id>       # Find idea brief card
python trello_ops.py approved_specs <idea_id>        # Get all approved specs for an idea
python trello_ops.py delete_all                      # Clear all cards (destructive)
```

## Environment

Copy `.env.example` to `.env` and populate:
- `TRELLO_API_KEY` / `TRELLO_API_TOKEN` — from https://trello.com/power-ups/admin
- `ANTHROPIC_API_KEY` — from https://console.anthropic.com/

`board_config.json` is generated by `setup_board.py` and required at runtime. It is not committed.

## Architecture

The system uses a **Trello board as both state store and dashboard**. Each Trello list is a pipeline stage; cards are work items that move left-to-right as agents process them.

### Orchestration model: Claude Code as orchestrator

**Claude Code (this tool) IS the orchestrator.** It drives the entire pipeline by:
1. Reading board state via `trello_ops.py`
2. Spawning Agent sub-agents for each pipeline stage's work (using the `agents/*.md` prompts as context)
3. Attaching agent outputs as files to Trello cards (one artifact file per card, replaced on each stage)
4. Moving cards between stages and updating labels
5. Detecting pipeline completion (all cards in Done)

This approach gives agents access to full Claude Code context, memories, and tools — unlike the legacy `orchestrator.py`/`agent_runner.py` which ran agents as external subprocesses.

### Pipeline stages (in order)

| Stage | Agent prompt | Notes |
|---|---|---|
| Intake | `intake_analyst.md` | Structures raw idea into an Idea Brief |
| Decompose | `feature_decomposer.md` | Splits idea into N feature cards; original card is archived to Done |
| Spec Write | `spec_writer.md` | Writes a Feature Spec; can receive bounce feedback |
| Spec Review | `spec_reviewer.md` | Approves or bounces back to Spec Write |
| Assemble | `prd_assembler.md` | Waits until all features are in Assemble, then builds PRD |
| Final Review | `final_reviewer.md` | Approves or bounces individual specs back to Spec Write |
| Done | — | Terminal state |

### Key modules

- **`trello_ops.py`** — CLI tool for all Trello board operations. Outputs JSON to stdout. Used by Claude Code orchestrator to read/write board state.
- **`setup_board.py`** — one-time board creation; writes `board_config.json`.
- **`agents/*.md`** — system prompts providing role context for each pipeline stage. Used as instructions when spawning Agent sub-agents.
- **`trello_client.py`** — async Trello API client using curl subprocesses (used internally by `trello_ops.py`).

### Orchestration protocol

When Claude Code processes a pipeline stage:

1. **Read board state**: `python trello_ops.py status` and `python trello_ops.py cards "<stage>"`
2. **Add progress label**: `python trello_ops.py add_label <card_id> "In Progress"`
3. **Spawn Agent sub-agent**: Pass the stage's system prompt + card content + idea brief as context. Run in background for parallelism.
4. **Attach output**: Write agent output to a local file, then `python trello_ops.py attach <card_id> <file>` — this replaces any prior attachment, so each card always has exactly one artifact file. Use comments only for lightweight notes (e.g., review decisions, bounce reasons). **Exception**: Spec Review and Final Review post their decisions as comments (not attachments), preserving the spec/PRD artifact on the card.
5. **Update label**: Remove "In Progress", add "Approved" or "Needs Revision"
6. **Move card immediately**: `python trello_ops.py move <card_id> "<next stage>"` — don't batch, move as soon as processed
7. **For bounces**: Move card back to the bounce target stage with "Needs Revision" label
8. **For archived cards** (e.g., idea card after Decompose): Remove "In Progress" label and add "Approved" before moving to Done
9. **Final deliverable**: When the PRD is approved, attach it to the **original idea card** in Done — this is the pipeline's output. Feature cards are intermediate work products and stay in Done with their individual specs.

### Board labels (progress indicators)

| Label | Color | Meaning |
|---|---|---|
| In Progress | Blue | Agent is currently processing this card |
| Approved | Green | Card passed review |
| Needs Revision | Purple | Card was bounced back with feedback |
| Blocked | Black | Card exceeded iteration limit |
| P0 - Must Have | Red | Priority label |
| P1 - Should Have | Orange | Priority label |
| P2 - Nice to Have | Yellow | Priority label |

### Card metadata

Workflow state is embedded in each card's description as an HTML comment:
```
<!-- workflow:{"idea_id":"idea-1234","iteration":0,"type":"idea|feature"} -->
```
`idea_id` links feature cards back to their parent idea throughout the pipeline.

### Parallelism strategy

- **Spec Write**: Launch all feature card writers in parallel (background agents)
- **Spec Review**: Launch all reviewers in parallel after all specs are posted
- **Bounced cards**: Can be pipelined — move revised spec to review immediately without waiting for all revisions
- **Assemble**: Runs once all specs reach the Assemble stage
- **Single-card stages** (Intake, Decompose, Final Review): Run sequentially
