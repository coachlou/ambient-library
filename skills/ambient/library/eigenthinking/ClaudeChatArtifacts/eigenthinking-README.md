# Eigenthinking
### Turn How You Think Into What You Own

---

## What It Is

Eigenthinking is a nine-step AI-powered framework for knowledge entrepreneurs who want to transform their tacit expertise into named, packaged, publishable Intellectual Property.

The name comes from linear algebra: eigenvectors are the natural transformation axes of a system — the directions that produce maximum movement per unit of input. Every expert thinker has cognitive eigenvectors too. The patterns of inquiry that consistently produce disproportionate insight. The questions they instinctively ask that others don't. The diagnoses they reach that their peers miss.

Most experts never name these. They operate on instinct, can't fully transfer the process, and watch AI flatten the value of everything they know.

Eigenthinking changes that.

**The core equation:** `UP → IP`  
Your Unique Process, made explicit, becomes Intellectual Property.

---

## The Problem It Solves

When AI is asked a question without a structured expert process behind it, it returns the modal response — the most probable answer from the center of its training distribution. Competent. Inoffensive. Indistinguishable from what everyone else got when they asked the same question.

For knowledge entrepreneurs, that's the problem. Modal answers are shared answers. Your client can get the same answer. Your competitor can get the same answer. The consultant who discovered AI last Tuesday can get the same answer.

Eigenthinking breaks this at the source. When a methodology is built on structural diagnosis, genuine reframe, and domain pattern recognition, the AI is no longer generating from the center of the distribution. It's generating from a region shaped by the expert's specific thinking. The output carries their fingerprints because their thinking defined the terrain.

**The result:** IP that can't be replicated without the expert's cognitive eigenvectors.

## The Nine-Step Process

```
SIGNAL → DRILL → REFRAME → NAME → BUILD → STRESS TEST → COMPRESS → LIFT → TEACH
```

| Step | What It Does |
|------|-------------|
| **1. SIGNAL** | Find the structural friction worth following — not the symptom, the signal |
| **2. DRILL** | Diagnose the mechanism, not the surface. Keep asking "what causes this?" until the answer is structural |
| **3. REFRAME** | Build the new lens that makes solutions visible and changes what questions are worth asking |
| **4. NAME** | Make the concept portable, ownable, and curiosity-gap-creating |
| **5. BUILD** | Turn the reframe into a sequential, testable, named methodology with quality gates |
| **6. STRESS TEST** | Find the honesty gap before the market does — Red Team, Edge Case, Honesty Check |
| **7. COMPRESS** | Prove the thinking survives reduction — one sentence, one paragraph, full document |
| **8. LIFT** | Move one abstraction level up: name the principle the methodology instantiates |
| **9. TEACH** | Produce the artifact that demonstrates the methodology, not just describes it |

Each step has a clear completion signal. The skill doesn't advance until the signal is present.

## What It Produces

A complete Eigenthinking session delivers:

- **A named concept** — with a curiosity-gap name that hints at the mechanism
- **A structured methodology** — sequential, testable, named at every level, with quality gates
- **A compression** — one-sentence and one-paragraph versions
- **A lift** — the principle the methodology instantiates (one level above the methodology)
- **A teaching artifact** — article, SOP, framework document, course module, or skill
- **A cognitive fingerprint** *(optional)* — a reverse-engineered profile of the expert's thinking patterns, usable as IP in its own right

## Evals

The skill ships with a companion evals file (`eigenthinking-evals.json`) that is **not bundled inside the `.skill` package**. This is by design — evals are development tooling, not runtime instructions. The `.skill` file is what gets loaded into context when the skill runs. The evals file is what you use to test and iterate on performance.

**Five test cases covering the full entry-point range:**

| Eval | Scenario |
|------|----------|
| 1 | Raw friction — "clients understand but don't implement" |
| 2 | Partially-named intuition — user has a label but no methodology |
| 3 | Generic IP request — "help me package my expertise" |
| 4 | Pure production — methodology exists, needs an artifact |
| 5 | Cognitive fingerprint — reverse-engineer my thinking patterns |

Each case includes 6–9 specific assertions that test whether the skill is behaving as designed, not just producing plausible-sounding output.

**To use:** Point the skill-creator at `eigenthinking-evals.json` and run the eval loop against any version of the skill you want to test. The five cases are designed to catch drift across the full skill surface — not just one scenario.
